## Q. 프록시 서버와 프록시 패턴의 차이점은 무엇인가요?

### 핵심 답변
프록시 패턴은 소프트웨어 설계 패턴으로 객체 간 접근을 제어하고, 프록시 서버는 네트워크 아키텍처 개념으로 클라이언트와 서버 사이에 위치해 요청을 중계합니다. 이름이 같지만 다른 계층의 개념입니다.

### 상세 설명

**공통점:**
- 실제 대상 앞에 위치하여 요청을 가로채는 "대리자" 역할
- 클라이언트는 프록시가 있는지 몰라도 됨
- 접근 제어, 캐싱, 로깅 등 부가 기능 추가

**차이점:**
| | 프록시 패턴 | 프록시 서버 |
|---|---|---|
| 적용 대상 | 소프트웨어 객체 | 네트워크 요청 |
| 위치 | 애플리케이션 코드 내부 | 클라이언트-서버 사이 (인프라) |
| 구현 | 디자인 패턴 | 실제 서버 소프트웨어 (nginx, CloudFlare 등) |
| 목적 | 객체 접근 제어 | 네트워크 요청 중계/제어 |

**프록시 서버의 역할:**
1. **포워드 프록시**: 클라이언트 앞에 위치, 클라이언트 신원 숨김 (VPN 유사)
2. **리버스 프록시**: 서버 앞에 위치, 서버 신원 숨김, 로드 밸런싱

```
[포워드 프록시]
Client → [Forward Proxy] → Internet → Server

[리버스 프록시]
Client → Internet → [Reverse Proxy (nginx)] → Server1, Server2, Server3
```

**디자인 패턴 관점의 유사성:**
프록시 서버도 원본 서버에 대한 접근을 제어한다는 점에서 디자인 패턴의 프록시와 동일한 개념적 구조를 가집니다.

### 꼬리 질문
- 포워드 프록시와 리버스 프록시의 차이는 무엇인가요?
- CDN은 프록시 서버의 일종인가요?

### 실무 사례
개발 환경에서 CORS 해결을 위해 webpack-dev-server의 `proxy` 설정을 사용하는 것이 포워드 프록시의 개발 활용 사례입니다.
