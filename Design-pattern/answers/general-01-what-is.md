## Q. 디자인 패턴이란 무엇이며, 왜 사용하나요?

### 핵심 답변
디자인 패턴은 소프트웨어 개발에서 자주 발생하는 문제에 대한 검증된 해결책의 템플릿입니다. 코드 재사용성을 높이고, 팀 간 의사소통을 원활하게 하며, 검증된 설계를 빠르게 적용하기 위해 사용합니다.

### 상세 설명

**GoF(Gang of Four) 디자인 패턴 분류:**
| 분류 | 설명 | 예시 |
|------|------|------|
| **생성(Creational)** | 객체 생성 방식 | 싱글톤, 팩토리, 빌더, 프로토타입 |
| **구조(Structural)** | 클래스/객체 조합 방식 | 어댑터, 데코레이터, 프록시, 컴포지트 |
| **행동(Behavioral)** | 객체 간 상호작용 | 옵저버, 전략, 이터레이터, 커맨드 |

**왜 사용하나요?**

**1. 검증된 해결책**
- 수많은 개발자들이 검증한 솔루션
- 바퀴를 다시 발명하지 않아도 됨

**2. 공통 언어 (의사소통)**
```
❌ "이 클래스는 인스턴스를 하나만 만들고 전역으로 접근하도록 해주세요."
✅ "싱글톤으로 구현해주세요."
```

**3. 재사용성 및 유지보수성**
- 패턴을 알면 새 코드베이스에서도 구조를 빠르게 이해
- 변경에 유연한 구조 제공

**4. OOP 원칙 자연스럽게 적용**
- 패턴을 따르면 SOLID 원칙이 자연스럽게 준수됨

**주의사항:**
- 패턴 적용이 목적이 아닌 수단
- 모든 문제에 패턴이 필요하진 않음 (Over-engineering 주의)

### 실무 사례
팀 코드 리뷰에서 "이 부분은 팩토리 패턴을 적용하면 좋겠다"고 말하면 즉시 의미가 전달됩니다. 패턴 이름이 공통 언어 역할을 합니다.
