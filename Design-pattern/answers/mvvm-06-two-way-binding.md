## Q. 양방향 데이터 바인딩의 장단점은 무엇인가요?

### 핵심 답변
양방향 바인딩은 UI와 데이터를 자동 동기화하여 개발 편의성이 높지만, 대규모 앱에서 상태 추적이 어렵고 성능 이슈가 발생할 수 있습니다.

### 상세 설명

**장점:**

**1. 개발 편의성 (코드 간결)**
```html
<!-- 양방향 바인딩 (Vue v-model) -->
<input v-model="email" />
<input v-model="password" />
<button @click="login">로그인</button>

<!-- 단방향 (React 방식) — 더 많은 코드 -->
<input value={email} onChange={e => setEmail(e.target.value)} />
<input value={password} onChange={e => setPassword(e.target.value)} />
```

**2. 폼 처리에 최적**
복잡한 폼의 경우 수십 개 필드를 v-model로 간단히 처리

**3. 실시간 동기화**
사용자 입력이 즉시 데이터에 반영되어 반응형 UI 구현 용이

**단점:**

**1. 상태 추적 어려움**
```
데이터 변경의 출처를 파악하기 어려움
A 컴포넌트에서 변경? 사용자 입력? 부모에서 전달?
```

**2. 성능 이슈**
```javascript
// 모든 키 입력마다 반응형 업데이트 발생
// 불필요한 렌더링이 많을 수 있음
data: { searchText: '' } // 타이핑마다 변경
```

**3. 디버깅 복잡성**
Angular 2+ 이전: 양방향 바인딩의 연쇄 업데이트(Digest Cycle)로 인한 무한 루프 가능

**4. 예측 불가능한 상태**
```
컴포넌트 A가 공유 데이터 변경
→ 컴포넌트 B도 자동 변경
→ 컴포넌트 C도 변경
→ 누가 변경했는지 추적 어려움
```

**React가 단방향을 선택한 이유:**
- 데이터 흐름 예측 가능성 (항상 위→아래)
- 복잡한 앱에서 상태 관리 용이
- 시간여행 디버깅 가능 (Redux DevTools)

### 꼬리 질문
- Flux/Redux가 단방향 데이터 흐름을 강제하는 이유는?
- Vue에서 컴포넌트 간 양방향 데이터 흐름을 제한하는 방법은?

### 실무 사례
Vue 3에서도 props는 단방향(`props down`)이고, 자식이 부모 데이터를 변경하려면 `emit` 이벤트를 사용(`events up`)하도록 합니다. v-model은 컴포넌트 내부에서만 사용을 권장합니다.
